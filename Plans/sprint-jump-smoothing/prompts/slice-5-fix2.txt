## Slice 5 Fix 2 — Air-Control Reversal: Facing Gate

### What we know
`JumpAirControl_FullReverseInput_RetainsAtLeastSeventyPercentOfForwardTravel` fails:
- Zero-input forward travel: 0.816m (baseline)
- Reverse-input forward travel: 0.002m (test result)
- Required: ≥ 70% of baseline = ≥ 0.571m

The character almost completely stops mid-air with reverse input held.

### Root cause (diagnosed)
`UpdateFacingDirection()` is called every FixedUpdate regardless of whether the character is airborne
mid-jump. When the player holds `Vector2.down` (backwards) during a jump:

1. `UpdateFacingDirection` tries to rotate the character to face backwards
2. `BalanceController` sees the character leaning backward (facing opposite to travel)
3. `ApplyLeanBraking()` fires — it applies -velocity * brakeCoeff based on lean angle
4. This kills forward carry almost entirely, hence 0.002m

The lean braking path is **input-independent** (physics-driven from lean angle), but the lean angle is
**input-driven** because `UpdateFacingDirection` is pointing the character backward.

### The fix
Gate `UpdateFacingDirection` (or the yaw torque it produces) behind `_recentJumpAirborne`:

```csharp
// In FixedUpdate, where UpdateFacingDirection is called:
if (!_recentJumpAirborne || _balance.IsGrounded)
{
    UpdateFacingDirection(_currentMoveInput);
}
```

This means during a jump arc, the character holds its facing from launch — no mid-air yaw rotation.
The air-control correction force can still trim lateral position (slice 4 feature), but facing won't
fight the balance controller.

**Important:** Only gate the full yaw update. Do NOT gate `_hasReceivedMovementInput` or any other
state that feeds into the correction force path.

### Verify the fix makes sense
After gating, check `UpdateFacingDirection` to understand what it does — confirm it's the yaw/rotation
driver and not something else:
```
Select-String -Path "Assets\Scripts\Character\PlayerMovement.cs" -Pattern "UpdateFacingDirection"
```

If `UpdateFacingDirection` also sets `_hasReceivedMovementInput = true`, that's fine — that flag feeds
the correction force, which is gated separately on `_recentJumpAirborne && !IsGrounded`.
