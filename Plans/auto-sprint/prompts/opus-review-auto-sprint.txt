## Testing philosophy (CRITICAL)
Tests must measure **what the game looks like when it's working**, not implementation details.

Good test: "Character reaches sprint speed within N seconds of sustained movement" — observable outcome
Bad test: "_sprintHeld is true after 1.2 seconds" — testing an internal variable

Ask for every proposed test:
- "If this feature was completely broken but internal variables still returned correct values, would this test catch it?"
- "Is this testing the outcome (what the player experiences) or the input (what the code does internally)?"

## The plan

# Plan 08 — Auto-Sprint

Remove the sprint button entirely. Character walks when you first move, auto-ramps to sprint after ~1.2s of sustained movement. Stopping resets the timer. "Land into a run" — landing from a sprint-speed jump flows directly into a run without stumble.

**Slice 1 — Auto-sprint timer**: Replace `_sprintHeld = sprint button` with timer-based logic in `PlayerMovement.cs`. `_autoSprintDelay = 1.2s`, small reset grace `_autoSprintResetOnStopDelay = 0.1s`. `SetSprintInputForTest` override still works.

**Slice 2 — Land into a run**: Store `_wasSprintingAtLaunch` at jump wind-up. On landing, if `SprintNormalized > _landIntoRunSprintThreshold (0.9)`: skip horizontal damping, set auto-sprint timer immediately to delay value. Walk-jumps unaffected.

**Slice 3 — Tests**: `AutoSprintTests.cs` — walk ramps to sprint after delay, stopping resets timer, land-into-run maintains speed, walk-jump does not land at sprint speed.

### 1. Results-based testing (most important)
For every proposed test:
- Is it testing an observable outcome or an internal variable?
- Would it catch a regression where the feature is visually/functionally broken?

- `AutoSprint_LandIntoRun_MaintainsSpeed` — speed measurement is good; what threshold is realistic?

### 2. Root cause vs patchwork
- Is "set timer to delay value on landing" actually the right fix for land-into-run, or is there a cleaner underlying model?
- Are there cases where the auto-sprint delay should be longer or shorter (e.g. after a fall, after getting up, after being knocked down)? Should sprint reset on fall/knockdown?
- Does the 0.1s reset grace feel right, or will it cause weird re-sprinting after quick direction changes?

### 3. Slice sizing
- Only 2 logic slices + 1 test slice — is this right-sized or can slice 1 and 2 merge safely?
- Is the Input Actions file change (removing sprint button binding) safe to do in slice 1, or should it wait until tests are green?

### 4. Gaps
- What happens to sprint state during jump wind-up and airborne? Currently `_sprintHeld` drives jump behaviour — does auto-sprint interact with jump code correctly?
- What about the existing `SprintRampFrames = 500` test constant — will auto-sprint break any existing tests that manually ramp sprint?
- Should there be a minimum movement distance before sprint activates, not just time? (Prevents sprinting in place if somehow move input is held but character isn't moving)
- Any UI/feedback implications — currently sprint button gave tactile feedback; auto-sprint is invisible. Does the player need any visual cue that sprint is active?

### 5. Ordering

- Could the auto-sprint timer cause unexpected sprint activation in edge cases (e.g. character being pushed by physics but player isn't inputting movement)?

### 7. Physics correctness
