# Copilot Agent Instructions - PhysicsDrivenMovementDemo

> Thin workspace entry point for GPT-5.4 agents. Use this file to route into the right durable doc instead of loading the whole repo surface up front.

## Project Summary

Physics-driven active-ragdoll obstacle course game in Unity 6. Characters are always ragdolls; locomotion, balance, recovery, and knockdown all come from forces, joints, and stateful runtime control. Game vision: handcrafted Wipeout/Only Up-style course, no respawns, physics chaos as the core loop. See `Plans/GAME_VISION.md` for the full vision doc. NGO networking is in the project but multiplayer is a future add-on.

## Context Budget Rules

- If a doc exposes `Quick Load` or `Read More When`, read only those sections first and continue deeper only when the task actually needs it.
- Prefer the active parent plan or roadmap chapter and the freshest artifact summary before raw logs, raw XML, or older child docs.
- Do not read every roadmap chapter by default. Open only the chapter that owns the task.
- Do not read `LOCOMOTION_BASELINES.md` end to end unless the task is baseline or regression comparison work.
- For test failures, start with `TestResults/latest-summary.md`, then the relevant XML, and only then the full Unity log if needed.
- For live Unity editor tasks, inspect the relevant editor builder scripts, scenes, prefabs, or menu-entry points before choosing shell commands or direct asset edits.

## Repo-Specific Guardrails

- Use `vscode_askQuestions` before most non-trivial planning or implementation work unless the request is already fully explicit.
- Keep the active parent plan or roadmap chapter current as soon as subtasks, blockers, evidence, or next steps change.
- Treat `.github/instructions/*.instructions.md` as thin routing shims and the linked `Plans/` docs as the canonical resumable execution record.
- Keep classes small: aim for roughly 300-350 lines of logic, stay normally under 500 total lines, and refactor before 600.
- Use `Tools/Run-UnityTests.ps1` as the authoritative unattended test path. Run EditMode and PlayMode sequentially and finish with the smallest relevant regression slice.
- Prefer outcome-based PlayMode verification for physics behavior over assertions that only inspect internal inputs.
